# Agent Execution Protocol

本文件是项目级硬约束。所有任务必须按“目标契约、证据账本、验收门禁、受限汇报”执行，避免目标漂移、证据降级和过度闭环。

## 1. 任务开始：目标契约

开始复杂任务前，先建立任务契约。用户已经给出计划时，必须逐条保留并转成 checklist，不能重写成简化版。

任务契约必须包含：

- 目标：本次真正要达成什么。
- 非目标：本次明确不做什么。
- 验收条件：什么证据能证明目标完成。
- 必须证据：代码、测试、数据库、日志、真实 UI、截图、人工确认等。
- 禁止性语言：未满足验收条件前，禁止说“完成”“已解决”“全部”“为 0”“没问题”“清完”。

Checklist 状态只能使用：

- 未开始
- 进行中
- 已验证
- 未验证
- 阻塞

## 2. 执行中：证据账本

每个关键结论必须绑定证据来源。没有证据的内容只能标为推断或未验证。

证据分级：

- 已验证事实：由测试、真实 UI、数据库查询、日志、截图或人工确认直接证明。
- 观察结果：实际看到的状态，但覆盖范围有限。
- 推断：基于代码或日志推理，不能当作事实。
- 未验证：尚未执行验证路径。

弱证据不能支撑强结论：

- 关键词 SQL 只能证明“这些关键词未命中”，不能证明“全库无污染”。
- 单测通过只能证明测试覆盖的行为，不能证明真实 UI 流程无问题。
- 代码静态分析不能替代真实 UI 自查。
- 局部样本成功不能说成全量成功。

## 3. 验收门禁

只有核心验收项全部具备强证据，才能汇报“完成”。

如果用户要求“自查”“审计”“确认”“验证”“彻底检查”，必须执行真实验证路径；不能用代码推断、关键词扫描或局部日志替代。

如果仍有未验证范围，最终状态必须降级为：

- 部分完成
- 待验证
- 已完成已知范围，未完成全量验证

高风险任务必须设置停止线：

- 真实用户消息发送
- 数据库删除或迁移
- 线上状态变更
- 支付、账号、权限操作
- 无法回滚或难以观察的副作用

没有明确证据和必要确认，不得越过停止线。

## 4. 汇报格式

所有阶段性和最终汇报必须区分进度与闭环。

默认结构：

- 结论：当前能被证据支持的最强结论。
- 已完成什么：实际做过的动作。
- 证据：支持结论的文件、命令、测试、数据库查询、日志、截图或真实 UI 记录。
- 未完成/未验证：尚未证明的范围。
- 下一步：如果未闭环，下一步验证或修复动作。

禁止把“做了什么”写成“证明了什么”。

## 5. 最终自检

最终回复前必须逐项自检：

- 原始目标是否逐条满足？
- 每个“已完成”是否有证据？
- 有没有未验证范围？
- 有没有用弱证据支撑强结论？
- 有没有把行动进度说成问题闭环？
- 有没有因为已经做了很多就提前收尾？

任一答案不满足时，最终汇报必须明确降级，并列出缺口。

## 6. 复用方式

将本文件复制到任一项目根目录，作为该项目的 `AGENTS.md`。

所有项目默认使用本协议；项目特有规则可以追加在本文件后面，但不得削弱以上门禁。
